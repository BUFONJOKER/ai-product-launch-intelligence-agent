Task: Auth Middleware – API Key & Secret Validation

This page outlines the implementation of the security layer for the FastAPI backend. By creating a centralized Auth Middleware, we ensure that only authorized requests can trigger expensive agentic workflows, while also validating that the necessary service keys (OpenAI, Firecrawl) are present before execution begins.

Overview

In a multi-agent system, your API acts as a gateway to high-cost LLMs and crawling services. The Auth Middleware performs two roles:

  1. Inbound Validation: Checking that the request from your Next.js frontend contains a valid "Application Secret."
  1. Service Readiness: Verifying that the backend environment has successfully loaded the external keys (Firecrawl/OpenAI) required to fulfill the request.

Description

We utilize FastAPI’s Security and HTTPBearer (or Custom Header) dependencies. This allows us to inject authentication checks directly into the route definitions, keeping the business logic clean while maintaining a "Secure by Default" posture.

Steps to Complete

  1. Security Config (core/security.py):
    • Define an API_KEY_NAME (e.g., X-Internal-Secret) and retrieve its valid value from environment variables.
  1. Dependency Implementation:
    • Create a validate_api_key function using FastAPI's Header or Security module.
    • Logic: If the header is missing or incorrect, raise an HTTPException with a 403 Forbidden status.
  1. Service Key Verification:
    • Implement a "Health Check" middleware or startup event that verifies the presence of OPENAI_API_KEY and FIRECRAWL_API_KEY.
    • If keys are missing, the server should fail to start (Fail-Fast) to prevent runtime crashes during agent execution.
  1. Route Integration:
    • Apply the security dependency to the main API router in app/api/v1/research.py:

      router = APIRouter(dependencies=[Depends(validate_api_key)])

  1. CORS & Secure Headers:
    • Configure the CORSMiddleware to strictly allow only your Next.js frontend domain and explicitly permit the custom auth headers.

Acceptance Criteria

  • Unauthorized Rejection: Any request sent without the correct X-Internal-Secret must return a 403 error immediately.
  • Header Transparency: The middleware should not interfere with standard HTTP methods (OPTIONS/CORS pre-flight).
  • Clear Error Messaging: If a service key (e.g., Firecrawl) is missing in the backend, the API should return a descriptive 500 Internal Server Error explaining the configuration gap.
  • Zero Latency Impact: The validation check must be a simple string comparison or hash check to ensure it doesn't slow down the request cycle.
  • Security Best Practices: Keys must never be hardcoded; the middleware must pull exclusively from app/core/config.py.
