Task: CORS Config – Secure Frontend-Backend Communication

This page outlines the implementation of Cross-Origin Resource Sharing (CORS) in the FastAPI backend. Because your Next.js frontend and FastAPI backend operate on different domains (or ports during development on WSL2), the browser's "Same-Origin Policy" will block API requests by default. This configuration creates a secure handshake that permits only your specific frontend to communicate with the LangGraph agents.

Overview

CORS is a security mechanism that uses additional HTTP headers to tell browsers whether a web application running at one origin has permission to access selected resources from a server at a different origin. In our stack, we must explicitly whitelist your Next.js domain while blocking all other unauthorized traffic.

Description

We will utilize FastAPI’s built-in CORSMiddleware. This middleware intercepts every incoming request and adds the appropriate Access-Control-Allow-* headers if the origin is trusted. It also handles Preflight (OPTIONS) requests, which the browser sends automatically before "complex" requests like those containing your custom auth headers (X-Internal-Secret).

Steps to Complete

  1. Origin Definition (core/config.py):
    • Define a list of allowed origins in your Settings class.
    • Include your local dev URLs (e.g., http://localhost:3000) and your production domain.
  1. Middleware Integration (main.py):
    • Import CORSMiddleware from fastapi.middleware.cors.
    • Apply the middleware to the FastAPI app instance.
  1. Security Parameter Tuning:
    • allow_origins: Set to your specific list (never use ["*"] in production).
    • allow_methods: Limit to the methods we use: ["GET", "POST", "OPTIONS"].
    • allow_headers: Must include standard headers AND your custom auth headers (e.g., ["Authorization", "Content-Type", "X-Internal-Secret"]).
    • allow_credentials: Set to True if you plan to pass cookies or session headers from Supabase.
  1. WSL2 Networking Check:
    • Ensure that if you are accessing the API from Windows while the server runs on WSL2, the IP mapping matches the allowed origin (often 127.0.0.1 vs localhost).
  1. Environment-Based Logic:
    • Configure the list of origins to change dynamically based on the APP_ENV variable (Development vs. Production).

Acceptance Criteria

  • Preflight Success: All OPTIONS requests from the browser must return a 200 OK with the correct CORS headers.
  • Unauthorized Rejection: Attempting to fetch the API from an unlisted domain (e.g., using curl from a random site) should result in a browser-level blocking or a CORS error.
  • Header Compatibility: The frontend must be able to send the X-Internal-Secret header without triggering a "Refused to set unsafe header" error.
  • Production Readiness: The production build must not contain localhost or wildcard () origins.
  • Persistence: The configuration must be loaded from your .env file via uv to avoid hardcoded domain strings.
