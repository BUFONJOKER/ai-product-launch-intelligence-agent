Task: FE-09: Data Fetching & State Management

This page outlines the implementation of the data synchronization layer between the Next.js frontend and the Supabase/LangGraph backend. We will utilize TanStack Query (React Query) to handle the complexities of server state, including caching, background refetching, and real-time status tracking for long-running agent threads.

Overview

In an AI-driven application, data is rarely static. The Data Fetching layer ensures that the UI stays in sync with the agent's progress. By using React Query, we avoid "Loading Hell" and provide a snappy experience where session history is instantly accessible, and report statuses (e.g., In Progress, Awaiting Approval, Completed) update automatically without manual page refreshes.

Description

The implementation focuses on two primary data streams:

  • Session History: A list of all research threads associated with the user, fetched from the Supabase Postgres tables.
  • Report Status & State: The real-time snapshot of a specific LangGraph execution, including the messages, specialist outputs, and current node position.

Steps to Complete

  1. Query Client Provider Setup:
    • Initialize the QueryClient and wrap the application in the QueryClientProvider within the root layout.
    • Configure global defaults, such as a 5-minute staleTime to reduce unnecessary database hits.
  1. Custom Hooks for Threads:
    • useThreads(): A hook to fetch the list of previous sessions for the sidebar.
    • useThread(id): A hook to fetch the full state and message history of a specific research task.
  1. Polling & Real-time Logic:
    • Implement Conditional Polling: If the thread status is running, set refetchInterval to 2 seconds. Once the status hits completed or interrupt, disable polling to save resources.
  1. Mutation Management:
    • useCreateThread(): Handles the initial POST request to start the LangGraph workflow.
    • useUpdateThread(): Handles user feedback during Human-in-the-Loop breakpoints.
  1. Prefetching & Optimistic Updates:
    • Implement prefetching on sidebar hover so that clicking a historical report feels instantaneous.
    • Use optimistic updates for "Rename" or "Delete" actions in the history list to provide immediate UI feedback.

Acceptance Criteria

  • State Sync: The "Thought Feed" and "Report" components must automatically reflect changes in the backend state without the user refreshing the browser.
  • Cache Management: Navigating back to a previously viewed report should load it instantly from the cache while a background refetch ensures the data is current.
  • Error Resilience: The system must implement "Retry with Backoff" for failed network requests, showing a "Reconnecting..." toast to the user.
  • Loading UX: Components must utilize the isLoading and isFetching states to display shadcn/ui Skeletons during initial transitions.
  • Resource Efficiency: Polling must strictly cease as soon as the agent reaches a terminal state (END) or a pause (Breakpoint).
