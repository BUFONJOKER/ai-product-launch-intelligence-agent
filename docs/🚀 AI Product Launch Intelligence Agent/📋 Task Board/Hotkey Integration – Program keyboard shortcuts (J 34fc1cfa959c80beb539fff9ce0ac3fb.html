Task: FE-07: Hotkey Integration

This page outlines the implementation of keyboard-driven navigation to enhance the workflow for power users. By mapping the J, K, and L keys to the intelligence dashboard's tabbed interface, users can cycle through analysis domains (Competitor, Sentiment, Metrics) without moving their hands from the keyboard.

Overview

In high-density data environments, reducing "mouse travel" significantly improves the user experience. Hotkey Integration treats the dashboard as a professional tool rather than a static webpage. We will implement a global listener that intercepts specific keystrokes to trigger state changes in the Tabs component.

Description

The shortcut mapping follows a logical sequence familiar to many professional software users (like Vim or Gmail):

  • J: Move to the Previous tab (to the left).
  • K: Move to the Next tab (to the right).
  • L: (Optional/Contextual) Lock or Expand the current view, or jump to the Collated Report.

For our specific three-tab setup (Competitor → Sentiment → Metrics), these keys will act as a "Carousel" navigator.

Steps to Complete

  1. Global Listener Implementation:
    • Create a custom React hook (e.g., useHotkeys) that attaches an event listener to the window object.
    • Implement logic to ensure hotkeys are disabled when the user is typing in an input field (e.g., the search bar) to prevent accidental tab switching.
  1. Tab State Mapping:
    • Define an array of the tab values in their visual order: ['competitor', 'sentiment', 'metrics'].
    • Create a "Next" and "Previous" function that calculates the new index, ensuring it wraps around (e.g., hitting 'K' on the Metrics tab returns to the Competitor tab).
  1. Visual Feedback (Osh):
    • Add a subtle "Active State" animation to the tab headers when triggered via keyboard to confirm the action.
    • Optionally show a small keyboard icon indicator or "toast" for first-time users to discover the shortcuts.
  1. Conflict Resolution:
    • Ensure the preventDefault() method is called on the event to stop the browser from performing default actions (like scrolling) when these keys are pressed.
  1. Accessibility (Aria):
    • Ensure that changing tabs via hotkeys correctly updates the aria-selected attribute and moves the browser focus if necessary for screen readers.

Acceptance Criteria

  • Responsiveness: The tab must switch immediately upon key press with zero perceptible lag.
  • Input Protection: Pressing 'J', 'K', or 'L' while the search bar is focused must result in the character being typed, not a tab switch.
  • Looping Logic: Navigation must be circular (Competitor ←→ Sentiment ←→ Metrics).
  • Cross-Browser Support: The hotkeys must function consistently across Chrome, Firefox, Safari, and Edge.
  • Discoverability: A "Keyboard Shortcuts" legend or tooltip should be available (e.g., visible when pressing ?).
