Task: RAG Integration – Vector Store Retriever

This page outlines the development of a Retrieval-Augmented Generation (RAG) tool. This enables your agents to supplement their global knowledge with private, internal company data stored in PDF manuals, project specs, and Notion documentation.

Overview

While web search tools provide external market data, the Vector Store Retriever focuses on the "internal truth." By indexing your company's proprietary documents, the agent can answer specific questions about internal processes, past project results, or specific product features that aren't available on the public web.

Description

The RAG tool utilizes a "Semantic Search" approach. Instead of looking for exact keyword matches, it uses vector embeddings to find information based on meaning. The tool retrieves the most relevant chunks of text from a vector database and provides them to the specialist nodes as high-fidelity context.

Steps to Complete

  1. Document Ingestion & Processing:
    • Implement a pipeline to load files (PDF, Markdown from Notion).
    • Split documents into smaller, manageable chunks (e.g., 500–1000 characters) to preserve context without overwhelming the LLM.
  1. Embedding Generation:
    • Select an embedding model to convert text chunks into numerical vectors.
    • Ensure the same model is used for both indexing and the user's search queries.
  1. Vector Store Setup:
    • Initialize a vector database (e.g., ChromaDB, Pinecone, or Supabase pgvector).
    • Upsert the embedded chunks into the database with relevant metadata (source, page number, date).
  1. Retriever Tool Logic:
    • Build the core function that accepts a natural language query.
    • Configure the tool to return the "Top K" most relevant results (e.g., the top 3–5 matching segments).
  1. Contextual Formatting:
    • Format the retrieved results into a clean string that clearly distinguishes between different sources so the agent can cite them.
  1. Agent Attachment:
    • Bind the retriever tool to the Launch Analyst or GTM Strategist nodes, allowing them to cross-reference internal goals with external market data.

Acceptance Criteria

  • Semantic Accuracy: The retriever must return relevant information even if the user's query uses different terminology than the document (e.g., "how we charge" should find "Pricing Structure").
  • Source Attribution: Every piece of retrieved information must include a reference to its source (e.g., Source: Q3_Strategy_Notion.md).
  • Latency Performance: The retrieval process should complete in under 1 second to maintain a fluid graph execution.
  • Data Privacy: The tool must strictly access only the authorized vector indices and not "hallucinate" information from outside the provided documents.
  • Integration: Specialist nodes must be prompted to prioritize the internal RAG data over general LLM knowledge when conflicts arise.
