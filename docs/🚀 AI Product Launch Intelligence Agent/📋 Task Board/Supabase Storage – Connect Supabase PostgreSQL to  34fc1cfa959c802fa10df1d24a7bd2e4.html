Task: Supabase Storage – Postgres Persistence Integration

This page outlines the implementation of the long-term storage layer for the application. While the LangGraph checkpointer handles short-term "thread" memory, we need a robust system to store the finalized, high-fidelity analysis reports. By utilizing Supabase (PostgreSQL), we ensure that every research task is archived, searchable, and retrievable for future comparisons.

Overview

In a production-ready AI system, data must be durable. We will configure a centralized database within your Supabase project to act as the "Source of Truth" for all research output. This setup allows the FastAPI backend to save structured JSON data and Markdown reports, while the Next.js frontend can fetch them instantly using the same thread_id.

Description

The integration focuses on two main storage patterns:

  1. State Checkpointing: Saving the granular, step-by-step history of the LangGraph execution.
  1. Report Persistence: Storing the final synthesized output from the Collator Node in a dedicated reports table for quick access and dashboard views.

Steps to Complete

  1. Database Schema Definition:
    • Create a reports table in Supabase via SQL or the Dashboard UI.
    • Columns: id (UUID), thread_id (UUID), user_id (UUID), report_json (JSONB), markdown_content (TEXT), and created_at (TIMESTAMPTZ).
  1. Connection Pooling (core/database.py):
    • Use psycopg2 or asyncpg to establish a connection pool.
    • Since you are working in WSL2, ensure the connection string uses the Transaction Mode or Session Mode URI provided by Supabase to optimize performance.
  1. Postgres Checkpointer Implementation:
    • Initialize the PostgresSaver provided by LangGraph.
    • Execute the checkpointer.setup() command to automatically create the necessary internal tables (checkpoints, checkpoint_blobs, checkpoint_writes).
  1. Data Upsert Logic:
    • Implement an "Upsert" function in the Collator Node that writes the final report to the reports table.
    • Ensure that if a report is refined (via Human-in-the-Loop feedback), the existing entry is updated rather than duplicated.
  1. Environment Variable Management:
    • Securely store your SUPABASE_DB_URL in the .env file and load it using the uvinstalled pydantic-settings.

Acceptance Criteria

  • Persistence Verification: After a research task finishes, the reports table in Supabase must contain a new entry with the correct thread_id.
  • State Recovery: If the FastAPI server restarts mid-analysis, the LangGraph must be able to resume from the last successful node using the database checkpoint.
  • Query Performance: Final reports must be retrievable by the frontend in under 300ms using indexed thread_id lookups.
  • Security: The database connection must utilize SSL, and the database user must have restricted permissions (CRUD only on specific schemas).
  • Sync with UV: The psycopg2-binary or asyncpg drivers must be correctly managed in the pyproject.toml.
